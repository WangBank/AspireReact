# 心魔录项目文档

## 1. 项目定位

心魔录的目标不是做一个泛交易系统，而是把“每天的账户状态、银证转账、股票流水和复盘笔记”沉淀成一套可持续回看的复盘资料。

项目围绕四类核心数据展开：

- 账户日资金
- 银证转账
- 股票交易与持仓快照
- 复盘笔记

当前系统在这四类数据之上，补齐了：

- 手机端券商截图 OCR 导入
- OCR 审计与原图回查
- 统一录入与统一列表
- 单票历史流水页
- 数据体检与异常提醒
- 统计分析、收益日历、盈亏榜
- 吾日三省吾身内容沉淀

## 2. 技术架构

| 层级 | 技术 |
| --- | --- |
| 前端 | React 19、TypeScript、Vite、MobX |
| 后端 | .NET 10、ASP.NET Core Web API、EF Core |
| 数据库 | PostgreSQL |
| 缓存 | Redis |
| OCR | RapidOCRSharpOnnx |
| 编排 | Lies AppHost |
| 部署 | Docker / Docker Compose |
| 日志 | Serilog |

架构职责划分：

- `frontend/` 负责页面、状态管理、表单、统计展示、移动端适配
- `Lies.Server/` 负责业务规则、数据落库、OCR、审计、统计汇总
- `Lies.AppHost/` 负责本地开发编排

## 4.5 Docker 部署

首次使用：

```bash
cp .env.docker.example .env.docker
docker compose --env-file .env.docker up -d --build
```

如果希望显式指定数据库镜像，可以在 `.env.docker` 中设置：

```env
POSTGRES_IMAGE=postgres:latest
```

注意：

- 现在不再把 PostgreSQL 大版本写死为 `17`
- PostgreSQL 18+ 的 compose 挂载路径应为 `/var/lib/postgresql`
- 如果本地已经存在旧版本 PostgreSQL 数据卷，直接切换到新的大版本镜像可能无法直接启动
- 生产或长期使用环境，升级前建议先做 `pg_dump` 备份或迁移

默认访问：

```text
http://localhost:5516
```

手写 `docker-compose.yml` 这条部署链路还会额外启动一个仅本机可访问的 Aspire Dashboard：

```text
http://localhost:18888/login?t=lies-dashboard-local
```

相关 `.env.docker` 配置：

```env
ASPIRE_DASHBOARD_IMAGE=mcr.microsoft.com/dotnet/aspire-dashboard:latest
ASPIRE_DASHBOARD_BIND_HOST=127.0.0.1
ASPIRE_DASHBOARD_PORT=18888
ASPIRE_DASHBOARD_FRONTEND_TOKEN=lies-dashboard-local
```

说明：

- 当前 Docker 方案中的 Dashboard 主要用于查看 `Lies.Server` 的 OTLP 日志、Trace 和指标
- 它是内网 / 本机访问用的观测面板，不建议直接暴露到公网
- 如果要看完整的 AppHost 资源编排页面，本地开发时仍然建议直接运行 `Lies.AppHost`

当前 `Lies.AppHost` 已经建模以下资源：

- `postgres`
- `redis`
- `server`
- `webfrontend`

Windows：

## 4.6 数据库备份恢复

如果手里已经有导出的 PostgreSQL 备份文件，可以直接恢复到当前运行中的 Docker Postgres。

项目内置恢复脚本：

- `scripts/db-restore.ps1`
- `scripts/db-restore.sh`

支持格式：

- `.sql`
- `.dump`
- `.backup`
- `.tar`

Windows PowerShell：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\db-restore.ps1 C:\Users\12243\Downloads\lies.dump lies
```

macOS / Linux：

```bash
bash scripts/db-restore.sh /path/to/lies.dump lies
```

恢复行为说明：

- 恢复前先启动 Docker 栈，手写 `docker-compose.yml` 链路和 `aspire-docker-up` 链路都支持
- 脚本会自动识别当前运行中的 `postgres` 容器，复制备份文件，断开活动连接，重建目标库，再执行恢复
- 目标库会被重建，原有数据会被覆盖
- 如果第二个参数不传，默认恢复到 `.env.docker` 中的 `POSTGRES_DB`，默认值为 `lies`
- 恢复凭据读取 `.env.docker` 中的 `POSTGRES_USER` 和 `POSTGRES_PASSWORD`
- 如果你走的是 AppHost Docker 发布链路，建议把 `.env.docker` 和 `.env.aspire-docker` 中的 PostgreSQL 用户名和密码保持一致
- 如果同一台机器上同时跑了多套 PostgreSQL 容器，优先停掉旧栈，避免恢复到错误容器

如果要使用“Docker + AppHost 参与资源服务”的部署方式，项目现在额外提供了官方 Docker Compose 发布链路。

先安装 Aspire CLI：

```bash
dotnet tool install --global Aspire.Cli --prerelease
```

准备独立环境文件：

```bash
cp .env.aspire-docker.example .env.aspire-docker
```

启动方式：

- macOS / Linux

```bash
bash scripts/aspire-docker-up.sh
```

- Windows PowerShell

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\aspire-docker-up.ps1
```

这条链路的发布态资源为：

- `dashboard`
- `postgres`
- `redis`
- `app`

默认行为：

- `app` 继续对外暴露 `5516`
- `dashboard`、`postgres`、`redis` 默认只绑定 `127.0.0.1`
- PostgreSQL `latest` 会按 18+ 规则挂到 `/var/lib/postgresql`
- AppHost 会自动生成 Docker Compose 产物，不需要手写维护资源关系

停止：

- macOS / Linux

```bash
bash scripts/aspire-docker-down.sh
```

- Windows PowerShell

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\aspire-docker-down.ps1
```

- 批量删除

交易列表额外规则：

- 区分持仓 / 清仓

## 6.4 统计页 `/statistics`

时间范围支持：

- 今日
- 本周
- 本月
- 本年
- 全部
- 自定义

分析内容覆盖：

- 区间总盈亏
- 银证转入 / 转出 / 净流入 / 当前总额
- 收益日历
- 单票累计盈亏排行
- 胜率分析
- 最大盈利区间 / 最大回撤区间
- 连赢连亏统计
- 净入金修正收益率
- 周期统计
- 做 T 专项分析
- 周 / 月 / 季度盈亏分布
- 资金使用率与波动率
- 板块轮动复盘
- 按卖出原因 / 情绪标签 / 交易标签的统计

## 6.5 股票历史页 `/stocks/:stockCode/history`

主要内容：

- 当前股票全部历史流水

- 周期盈亏与盈亏比
- 相关笔记联动

## 6.6 数据体检 `/health`

用于集中发现：

- 账户总资产与持仓 / 可用资金对不上
- OCR 识别存在缺字段但试图落库
- 交易记录持仓状态异常
- 历史脏数据导致的盈亏不一致

说明：

- 对历史手工录入数据不会强制要求补原图

## 6.7 OCR 审计 `/audits/imports`

用于：

- 查看识别批次
- 查看识别原图
- 对比识别结果与最终落库结果
- 回查哪次识别出错

## 6.8 笔记页面

全局笔记 `/notes/global`：

- 适合记录大盘、纪律、计划

心魔笔记 `/notes/stock`：

- 适合记录单票逻辑与复盘

吾日三省吾身 `/notes/reflection`：

- 预置外部原文
- 支持按句拆分与分栏维护

## 7. OCR 导入说明

## 7.1 当前支持的截图

- 同花顺手机端持仓整屏图
- 包含当日买卖、均价、收盘价的流水表
- 左侧账户汇总 + 右侧明细组合截图

## 7.2 识别流程

1. 上传图片
2. 识别日期
3. 解析账户汇总
4. 解析银证流水
5. 解析股票明细
6. 返回 warning、原图、回填数据
7. 用户确认后再保存

## 7.3 容错策略

当前已处理的重点问题：

- 手机端日期 OCR 粘连纠偏
- 表头别名与模糊匹配
- 证券代码与收盘价粘连修复
- 顶部当日盈亏与明细合计冲突时回退
- 缺少关键字段时阻止直接提交

## 7.4 使用建议

- 尽量保留完整日期与表头
- 不要裁掉左侧日期列
- 识别后先核对原图再保存
- 对做 T、半仓、清仓当天的记录重点检查卖出价与卖出量

## 8. 统计能力说明

## 8.1 区间概览

基于账户、流水、交易三套数据联合计算：

- 总盈亏
- 已实现盈亏
- 未实现盈亏
- 总入金 / 总出金 / 净流入
- 当前总额

## 8.2 周期统计

周期定义为：

- 从首次建仓开始
- 到清仓结束
- 中间允许加仓、减仓、做 T

统计结果包括：

- 周期数
- 已结束周期胜率
- 平均单周期盈利 / 亏损
- 平均持有天数
- 最大单周期盈利 / 亏损

## 8.3 净入金修正收益率

利用账户资金与银证转账，剔除资金转入转出干扰，更接近真实交易收益。

## 8.4 收益日历

支持：

- 按日
- 按月
- 按年

用途：

- 快速看哪几天、哪几个月最赚钱或最失控

## 8.5 标签分析

当前可从三个角度拆分交易：

- 卖出原因
- 情绪标签
- 交易标签

适合复盘：

- 哪类卖法最赚钱
- 哪种情绪最伤账户
- 哪种模式最稳定
